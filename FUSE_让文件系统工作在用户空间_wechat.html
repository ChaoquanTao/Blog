01

从一个具体需求说起

假设你接到一个需求：把远程服务器的目录挂载成本地文件夹，让同事直接用 vim、grep、文件管理器访问远程文件，就好像文件在本地一样。

需求很合理。怎么实现？

第一反应可能是"写个文件系统"。但你很快发现，传统文件系统跑在内核里——这意味着你得用受限的内核 C 开发，一个 bug 就能把整台机器搞宕机，还得为每个内核版本重新编译。你明明只是想做个 SSH 数据转发，却被迫先成为内核专家。

"

写一个自定义文件系统，一定要懂内核吗？有没有办法用普通的编程语言、在普通的用户态进程里实现文件系统逻辑，同时让 cat、ls、vim 完全无感地使用它？

这正是 FUSE 解决的问题。但要理解它的解法，得先理解它面对的约束。

02

文件系统为什么必须在内核里

内核空间与用户空间：那道不可逾越的墙

把电脑想象成一座城堡。最里层的内堡住着内核（kernel），它直接握着控制硬件的开关——硬盘、内存、网卡、CPU——在这一层，代码能对机器做任何事，也能让整台机器宕机。城堡外圈的民居住着普通应用程序：浏览器、你的脚本，这一层叫用户空间（user space）。

为什么要隔开？因为任何代码都可能出 bug，而 bug 在内堡里的后果是致命的。 浏览器崩了，如果它住在内堡、能直接碰硬件，一崩就拖垮整台机器；隔开之后，它崩了就崩了，操作系统负责回收清理，别的程序照跑。这道墙的全部意义就是隔离故障域。

// 内核态 / 用户态分层

RING 0 · 内核空间

硬件控制 · 文件系统 · 进程调度 · 内存管理

权力无限，一错全崩

━━━ 系统调用（syscall）━━━ CPU 硬件强制隔离 ━━━

RING 3 · 用户空间

浏览器 · 你的脚本 · vim · grep ...

崩了就崩了，不影响全局

这道墙是 CPU 在硅片层面物理保证的。x86 用"特权级（Ring）"实现：Ring 0 是内核模式，能执行任何指令；Ring 3 是用户模式，一大堆危险指令被硬件禁止。用户程序无法自己把档位从 Ring 3 改到 Ring 0——否则墙就形同虚设。

工程细节

升权只能从固定"正门"发生：系统调用（syscall）。你的程序执行一条 syscall 指令，CPU 一气呵成做两件不可分割的事——把权限档位升到 Ring 0，并强制跳转到内核早就设定好的入口地址。升权的同一瞬间，执行权就已经交到了内核手里，你没机会以 Ring 0 的身份跑自己的代码。

每一次系统调用，控制权都要"穿过护城河"一来一回。这种穿越是有成本的——后面 FUSE 的性能损耗，原因就在这里。

文件系统为什么必须住在内核

存储设备（硬盘、SSD）本身根本不懂"文件"。它只是一长条编了号的块（逻辑块地址，LBA），只会听"把第 N 号块的内容给我"。"文件""文件夹""文件名""权限"这些概念，全都是文件系统这层软件编造出来的幻觉——把文件名映射到散落在设备各处的块、维护目录树、记录元数据。ext4、NTFS、APFS 都是这层逻辑的不同实现。

关键点：这层逻辑跑在内核里（Ring 0）。 因为它要直接操作块设备硬件，而碰硬件是特权操作，只有 Ring 0 能干；而且文件系统是所有程序共享的关键基础设施，必须由内核统一看管。

当你想做一个"古怪文件系统"时，痛点来了

回到开篇的需求。类似的合理需求还有很多：把本地文件夹做成"往里拖文件就自动加密上传到云"；把一个 .zip 当成可浏览的目录。要实现它们，按传统做法，你得把这套逻辑写成内核代码、塞进 Ring 0。而这件事的痛点是：

开发地狱

内核里基本只能用受限的 C，没有你熟悉的网络库、加密库、高级语言。普通程序崩了重跑就行，内核代码崩了整台机器直接宕机，改一行验证一次的成本极高。

爆炸半径无限大

同样一个空指针 bug，发生在用户态只是"这个程序崩了，重开"；发生在 Ring 0 就是全机宕机，甚至默默写坏别人的内存。更糟的是，你的业务代码被攻破，因为它在 Ring 0，攻击者直接拿到整台机器的最高权限。

分发门槛高

装内核模块要 root，还要为用户的内核版本重新编译，内核接口跨版本还会变。

根本矛盾

文件系统逻辑必须活在内核里（才能碰硬件、才能被所有程序当文件用），但内核又是最不适合写普通业务逻辑的地方（语言受限、一错全崩、门槛高）。

一边拽着"必须在内核"，一边拽着"实在不该在内核"。
这就是 FUSE 要化解的矛盾。

03

FUSE 是什么

FUSE（Filesystem in Userspace）的核心用四个字总结：劈成两半。 内核里只留一个不含任何文件系统业务逻辑的"转发员"，真正的繁重工作全部交给一个普通的用户态进程去做。

// FUSE 核心架构

RING 3 · 用户空间

你的程序调用 read() / ls / cat ...

↓ syscall

RING 0 · 内核空间

VFS → FUSE 内核模块（空壳转发员）

不含文件系统业务逻辑，只做请求转发

↕ /dev/fuse（信使通道）

RING 3 · 用户空间

用户态 daemon（真正的大脑）

Python / Go / C++ 随便写，崩了只崩自己

三个角色

1

FUSE 内核模块（Ring 0 的"空壳"）：不含任何文件系统业务逻辑，唯一的工作是当二传手——把请求从内核 VFS 转发到用户态，再把结果转回来。逻辑极简，所以它不会爆炸。

2

/dev/fuse（信使通道）：内核空壳和 daemon 之间的通信被包装成一个设备文件。daemon 用最普通的 read() / write() 操作它。

3

用户态 daemon（Ring 3 的"真逻辑"）：真正的大脑，一个彻头彻尾的普通用户态程序——想用 Python 就 Python，网络库、加密库随便用。它崩了？只是这一个进程崩了，全机安然无恙。

此外还有 libfuse：一个用户态库，把"循环读 /dev/fuse、解析二进制协议、根据请求类型调用对应回调"这些样板代码封装好。有了它，你写文件系统时只需填几个回调函数——getattr、readdir、read、write 等等。

矛盾是怎么被化解的

✓ "必须在内核"

满足。内核里确实有 FUSE 模块，VFS 照常把它当文件系统对待，cat、vim、任何程序完全无感。

✓ "不该在内核写业务逻辑"

也满足。真正的业务逻辑根本不在内核，而在那个可以随便崩、随便用高级语言、不需要 root 编译内核的用户态 daemon 里。

FUSE 改变的不是"能否自定义文件系统"，而是"自定义这件事发生在哪"——它把爆炸半径从全机缩小到了一个进程。

三个设计决策的"不可省略性"

为什么内核里还非得留个空壳？

因为 cat/ls 这类程序的文件请求只会送进内核 VFS。必须有个内核侧的模块声明"这个挂载点归我管"，才能把请求转给 daemon。它复用了 VFS 这个"万能插座"，于是你的古怪文件系统自动兼容了世界上所有会读文件的软件。

为什么用 /dev/fuse 这个设备文件？

"一切皆文件"。daemon 用最熟悉的 read/write 即可，零学习成本。选它不是因为"快"——它甚至增加了开销——而是因为接口统一。

为什么要有 libfuse？

裸手操作 /dev/fuse、解析二进制协议的样板代码冗长繁琐且每个 FUSE 程序都一样。libfuse 封装好这些，你只写业务回调。

04

一次 read 是怎么走的

理解了架构，我们把同一次文件读取在两种文件系统里走一遍，体会 FUSE "劈分"的代价。

普通 ext4

① 你的程序（Ring 3）

调用 read()，模式切换进 Ring 0

↓

② 内核 ext4（Ring 0）

直接碰硬盘拿数据

↓

③ 程序拿到数据

模式切换回 Ring 3，结束

1 个进程 · 2 次模式切换 · 干净利落

FUSE

① 你的程序（Ring 3）

调用 read()，模式切换 ① 进 Ring 0

↓

② FUSE 空壳（Ring 0）

把请求挂到 /dev/fuse，阻塞你的程序

↓ 进程切换 Ⓐ 唤醒 daemon

③ daemon 被唤醒

read(/dev/fuse) 返回，模式切换 ② 回 Ring 3

↓

④ daemon 跑真逻辑（Ring 3）

SSH 拉数据 / 解密 / 计算... 然后 write 回 /dev/fuse
模式切换 ③ 进 Ring 0，④ 回 Ring 3

↓ 进程切换 Ⓑ 切回原程序

⑤ FUSE 空壳收到结果（Ring 0）

唤醒原程序，模式切换 ⑤ 回 Ring 3

↓

⑥ 你的程序拿到数据

普通 ext4 FUSE
涉及进程 1 个 2 个
进程切换 0 次 ≥2 次
模式切换 2 次 ≥5 次
数据拷贝 少 更多

为了把活儿外包给安全的用户态 daemon，每次操作凭空多出了进程切换和数倍的模式切换。这就是 FUSE 性能开销的来源。

这个开销什么时候可以忽略，什么时候不能

✓ 无感

sshfs 的瓶颈是网络延迟（几十毫秒），FUSE 那点切换开销（几微秒）淹没在里面根本看不见。

✗ 明显感知

对一个有成千上万文件的大目录执行 ls -l，每个文件都触发一次 getattr，每次都得绕一圈到 daemon。这就是 FUSE 文件系统 ls -l 常常很慢的原因。

缓解手段

内核页缓存能把读过的数据缓存起来，第二次读可能直接命中、不惊动 daemon；一次读一大块而非逐字节，把固定切换成本摊薄。但元数据操作的开销基本省不掉。

05

边界：真用起来会撞到的墙

daemon 崩了 / 被 kill

正在读这个文件系统的程序会收到错误（典型是 ENOTCONN 或 EIO）或卡住，可能留下需要 fusermount -u 强制卸载的"僵尸挂载点"。但全机存活，重启 daemon 即可恢复——这正是"爆炸半径从全城缩到一间民房"的具体兑现。

权限默认值

一个 FUSE 挂载默认只有挂载它的那个用户能访问，连 root 默认都看不到——这是防止恶意 FUSE 骗别人的安全默认值。要让别人也能访问，得显式开 allow_other。

不可信来源要警惕

daemon 是普通用户态进程，能看到所有经它的文件内容，还能对不同读取返回不同内容（制造 TOCTOU 竞态）。别随便去读不可信者挂载的 FUSE。

语义打折

mmap、某些 ioctl、文件锁的精确语义，因为多绕了一层用户态，支持有限或行为微妙。特别注意：direct_io 模式下 mmap 完全不可用。

06

意义：它改变了"谁有资格写文件系统"

FUSE 在真实世界养活了一大批东西，它们有一个共同点——都不是"在硬盘上存字节"，而是"把某种古怪的数据源伪装成文件"：

sshfs

数据源是网络，把远程目录挂成本地文件夹。

s3fs / rclone mount

数据源是云存储，把 S3、Google Drive 挂成本地盘。

gocryptfs / EncFS

你看到的是明文，落盘的是密文，加解密全在 daemon 里。

AppImage

运行时用 FUSE 把自己临时挂载解压出来跑——你可能用过却不知道。

为什么这件事重要？可以从三个层面理解：

1

门槛的本质是技能错配：FUSE 之前，"做个古怪文件系统"被锁死在内核专家手里。不是因为想法稀缺，而是因为要求的技能（内核 C、Ring 0 调试）和真正需要的技能（懂 SSH、懂加密、懂云 API）严重错配。

2

让"写文件系统"民主化了：它把所需技能从"内核开发"换成了"会写普通程序"。能力本身没变强，但能行使这个能力的人数暴增了几个数量级。世界因此多出了 sshfs、s3fs、gocryptfs——不是因为技术突破，而是因为门槛塌了。

3

伪装成文件 = 免费接入整个生态："文件"是操作系统生态里最通用的接口。你把数据源伪装成文件，vim、grep、Photoshop、python open()、文件管理器……所有会读文件的软件都自动会用它，你一行适配都不用写。

07

什么时候不该用 FUSE

✗ 追求极致性能的主力盘

ext4/NTFS 那种高频本地读写，FUSE 每次操作的切换开销在高频场景下非常明显。

✗ 元数据极密集的负载

海量小文件的频繁 stat / ls，getattr 的开销躲不掉。

✗ 强依赖精确底层语义

mmap、文件锁原子性等场景，语义打折可能导致难以排查的 bug。

FUSE 最适合的场景始终是：逻辑古怪、依赖外部库（网络/加密/云）、不追求极致性能、希望普通人也能写能装的文件系统。

FUSE 用"内核空壳 + 用户态灵魂"的劈分，把"写文件系统"从内核专家的高危特权，降格成普通开发者的日常；代价是每次操作多绕一圈的切换开销。

而它真正的价值，是让任意数据源借"文件"这个万能接口，瞬间接入整个软件生态。
