引子

先想象一个场景。你花重金租了一张 80GB 的 A100，把一个开源大模型部署上去。自己测试时体验丝滑——问一句、秒回一句。于是你信心满满地把它开放给整个团队。

几十个人同时涌进来，结果却出乎意料：有人转着圈等了半天才出字，更糟的是没过多久服务直接报错——显存不足。你赶紧打开监控，却看到一件怪事：这张卡明明还显示着不少空闲显存，为什么就服务不了几个人？

先停下来想一想

显卡账面上还有余量，实际却只能服务寥寥几个用户——瓶颈到底卡在哪？答案不在"算力够不够快"，而藏在一个容易被忽略的地方：显存是怎么被用掉、又是怎么被白白浪费掉的。

这篇文章，就从这桩"怪事"讲起，一步步拆开 vLLM 是怎么把同一张 GPU 压榨到能服务远多于平常的用户的。我们先从最基础的概念入手。

01

什么是"服务一个 LLM"

要看懂开头那桩怪事，得先弄清这套系统到底在干什么。大语言模型给它一段文本、它就返回另一段文本；所谓服务一个 LLM，就是把模型部署在机器上，让大量用户同时向它提问、并拿到回答。而负责把请求收进来、调度模型运行、再把回复发回去的那段软件，就叫服务引擎（serving engine）。

GPU 上的模型 · 干重活生成回复

LLM 跑在 GPU 上。GPU 算力强劲，但显存既有限又昂贵。显存每浪费一分，能服务的用户就少一分，成本也随之高一分。

服务 LLM 的本质目标：在一张 GPU 上，尽可能多、也尽可能快地服务用户。vLLM 正是为赢下这场博弈而生。

02

Prefill、Decode 与 KV cache

LLM 收到文字后，先把它切成更小的片段——叫做 token（词元）。token 可以是一个词、半个词或一个标点；为方便理解，下面近似把一个 token 当成一个词。

关键直觉：模型每"跑"一次，只能往外吐出 1 个新 token。要写出一整句话，就得让模型反复跑很多次，一次接一次地把词续上去。

▸ 第一阶段：Prefill（把 prompt 读进去）

假设 prompt 是 Tell me a joke，切成 4 个 token。模型把这 4 个 token 一次性、同时喂进去处理一遍，吐出回复的第 1 个 token。这一步只做一次。

接下来进入循环。每一步，模型都把"到目前为止的全部 token"重新看一遍，再吐出 1 个新 token 接到末尾。然后拿更长的序列再跑下一步，如此往复。

…… 每步比上一步长 1 个 token，直到吐出"结束符"或到达长度上限。

注意关键一点：decode 的每一步都要回头参考前面所有 token。这正是为什么 prefill 计算密集（一口气算完一长串），而 decode 是逐 token 慢慢挤出来的。

▸ 为什么需要 KV cache

decode 每一步都要"重新看一遍前面所有 token"。如果每次都把老 token 的内部计算从头再算一遍，会非常浪费，而且越往后越慢。于是有了关键优化：模型把每个 token 算过一次的中间结果（每一层、每个注意力头上的 Key / Value 张量）缓存起来，这份缓存就叫 KV cache。有了它，decode 每一步只需真正计算那 1 个新 token，前面的老 token 直接从缓存里读。

背景补充 · 注意力机制（熟悉 Transformer 可跳过）

模型处理一个 token 时要判断前面哪些 token 跟它有关，很像一次"软查询"：每个 token 产出三个向量——Query（我在找什么）、Key（我是关于什么的，相当于索引）、Value（我实际携带的信息）。当前 token 用自己的 Query 比对前面所有 token 的 Key，算出"该关注谁、关注多少"，再按权重汇总对应的 Value。

模型并行做很多份这种查询，每份叫一个注意力头：有的管语法搭配，有的弄清代词指代。模型有很多层，每层又有很多头，所以"每一层、每个头上"才都有一份 K/V。

例：The cat sat on the mat because it was tired. 处理到 it 时，它的 Query 与 cat 的 Key 最匹配，注意力权重集中到 cat，模型就此"读懂"了 it 指的是猫而非垫子。

为什么缓存 K/V 而非 Q：生成新 token 时是拿新 token 的 Query 去对照前面老 token 的 Key/Value。老 token 的 K/V 算过一次便不再改变，存下来即可；新 token 只需现算自己的 Q，再把 K/V 追加进缓存。

关键点

KV cache 会随回答变长而不断增长，且全部驻留在 GPU 显存里。以一个 7B 模型在 fp16 下为例，单个 token 的 KV cache 约几百 KB（具体取决于是否用 GQA：MHA 模型每 token 可达约 0.5MB，重度 GQA 会小很多）；上千 token 的长对话就要占掉数百 MB 乃至 GB 级——而一张卡的显存总共也就 24 / 40 / 80 GB。这份会不断膨胀的缓存，正是 vLLM 要解决的核心瓶颈。

03

问题：KV cache 吞噬显存，朴素方式还在浪费

模型本身就占掉一大块显存，剩下的空间才用来放所有正在服务请求的 KV cache。所以决定"一次能并发服务多少用户"的往往不是算力，而是留给 KV cache 的那点显存。服务 LLM 的真正瓶颈，从来不是算得快不快，而是显存管理。

朴素的服务引擎会做一件"看似稳妥、实则很浪费"的事：请求一进来，引擎并不知道回答会有多长，于是干脆按最坏情况（比如 2000 token）预留一整块连续显存。可绝大多数回答只有几十到一两百 token，剩下 1900 多 token 的空间就被白白锁住。

✗ 内部碎片 internal fragmentation

单个请求预留 2000、实际只用 50，剩下 1950 浪费在这个请求自己的内存块内部。

✗ 外部碎片 external fragmentation

多个请求各占一块，块与块之间散落零碎小空隙，单看任何一处都塞不下一个新请求。

朴素服务 · 每个请求预留一整块连续内存

请求 A

用50 为 2000 预留但浪费

请求 B

用120 为 2000 预留但浪费

请求 C

20 为 2000 预留但浪费

剩余空闲都是零碎小块 → 装不下任何新请求

现在，开头那桩怪事终于有了答案：那张卡账面上还剩不少显存，却服务不了几个人，谜底就是这两种碎片——空间不是不够，而是被锁死、被切碎了。这，正是 vLLM 要破解的局。

04

PagedAttention：把操作系统的分页搬到 KV cache

vLLM 是一个面向 LLM 推理的高吞吐服务引擎，核心是把 KV cache 的显存管理做到极致，主要靠两件事：PagedAttention（把 KV cache 拆成固定大小小块，用多少分多少）与连续批处理（每个 decode 步让请求动态进出，尽量不让 GPU 闲着）。

PagedAttention 借鉴操作系统的分页机制：不一次性连续分配，而是切成固定大小的页，按需分配、不要求物理相邻，再用页表记录每页落在哪里。vLLM 对 KV cache 做同样的事：

切块

把 KV cache 切成固定大小的 block，默认每块 16 个 token（可配置）。

块表 block table

每个请求维护一张块表，记录它逻辑上的第 0、1、2… 块分别落在物理上的哪一块。

按需取、随时还

需要更多 token 就从空闲池取一块，不要求物理相邻；请求一结束，占用的块立刻归还空闲池。

物理块池（颜色 = 归属）

B1B2B3B4B5B6B7B8B9

请求 A → B1, B4, B7 （按需 3 块）

请求 B → B2, B3 （按需 2 块）

请求 C → B5 （刚开始 1 块）

空闲块：B6, B8, B9

这里要分清两个概念：逻辑块是请求自己视角的"第 0、1、2 块"，编号连续；物理块是显存里实际占用的那一块，物理位置可散落在任何地方。

工程细节 · 这套机制是有代价的

注意力计算所需的 K/V 张量在显存里不再连续，普通的 attention CUDA kernel 没法直接用。现成的高性能 kernel（如 FlashAttention）正是建立在"一个序列的 K/V 在显存里连成一片"这个前提上写的——GPU 靠成千上万线程挨着读、按规整步长读才能榨干带宽。

PagedAttention 把 K/V 打散成物理四处散落的 block，等于抽走了原 kernel 提速的前提。于是 vLLM 必须重写一个 kernel：先查块表弄清每个逻辑位置对应哪个物理块，再去零散地址上取 K/V 做注意力，既兼容分页布局，又尽量不丢并行效率。这才是 PagedAttention 落地的真正难点——它远不止"加一个内存分配器"那么简单。

05

顺手的红利：PagedAttention 如何共享内存

一旦 KV cache 按块组织，内存共享几乎顺手就能拿到：两个不同请求，只要把各自块表里的某一项指向同一个物理块，就实现了共享。

相同前缀

很多请求都以同一段长 system prompt 开头（如"你是某汽车经销商的客服助手……"）。vLLM 把这段开头的 KV cache 只算一次、只存一份，所有请求的块表都指向那几块。

Beam search 束搜索

模型同时探索多条候选回答，它们共享同一个开头、只在后面才分叉。开头的块共享，分叉之后再各自分配。

连续批处理：让 GPU 一刻不空转

GPU 一次同时跑多个请求时效率最高，所以要做批处理。但最朴素的静态批处理是凑齐一批一起跑，必须等整批里最慢的那个结束，才能开始下一批。问题在于不同请求长度差别巨大：一个 20 token 就答完，另一个要 800 token。20 token 那个早早跑完，占的 slot 却得空等到 800 token 结束——这段空等纯属浪费。

连续批处理换了思路：在每一个 decode 步都重新决定这一批跑哪些请求——某个请求一结束，就立刻从等待队列拉一个新请求补进它的位置。

A 完 C 立即进，C 完 D 立即进 → GPU 不空转

连续批处理与 PagedAttention 是天作之合：前者一空出 slot，后者就立刻把对应 block 回收进空闲池，新请求随即拿到显存与算力，无缝接上。

07

OpenAI 兼容 API 与 vLLM 的收益

vLLM 不只是推理库，对外暴露的是一个 OpenAI 兼容的 HTTP 服务器——/v1/chat/completions 等接口一应俱全，支持流式输出、function calling。意义在于：用 OpenAI SDK 写好的现有代码，只要把 base_url 指向自己的 vLLM 地址，就能无缝切换到自家 GPU 上的开源模型，业务代码一行都不用动。

✓ 更高吞吐

PagedAttention 省显存、连续批处理省算力，二者叠加，很多场景下相比朴素引擎能拿到 2–4 倍甚至更高的吞吐提升（数字来自 SOSP '23 原始论文，实际幅度取决于工作负载）。

✓ 更高 GPU 利用率 / 更低单请求成本

显存和算力都能接近打满；一张卡服务更多用户，分摊到每个请求的硬件成本随之下降。

✓ 极低接入成本

一套 OpenAI 兼容 API，现有代码几乎拿来即用。

✗ 代价 · trade-off

连续批处理叠加大 batch，通常意味着单请求延迟（尤其 TTFT 和 TBT）上升——你的请求得和别人的一起排队调度。生产部署要在吞吐与延迟之间权衡，天下没有免费午餐。

08

横向对比：SGLang、TensorRT-LLM、LMDeploy

vLLM 并不是唯一选择。截至 2026 年，生产环境里常见的同类方案还有以下几个。

TensorRT-LLM NVIDIA

优势：官方出品，kernel 优化做到 PTX/SASS 层级，H100/H200/B200 上裸吞吐往往是天花板；in-flight batching、paged KV cache、speculative decoding、FP8/INT4/INT8 量化全支持。

劣势：需 AOT 编译，耗时长、新模型适配慢、调试门槛高，且只支持 NVIDIA 硬件。

适用：模型稳定不常换、硬件锁定 NVIDIA、对延迟极敏感的场景。

SGLang Berkeley 等

优势：核心创新 RadixAttention——用基数树做 KV cache 前缀共享，比 vLLM 更激进；agent、多轮对话、RAG 等前缀大量重复的场景命中率和性能明显更优；内置面向 LLM 程序的 DSL。

劣势：社区比 vLLM 小，硬件后端覆盖窄，冷门模型适配慢。

适用：以 agent / RAG / 多轮对话为主、前缀复用率高的工作流。

LMDeploy 上海 AI Lab

优势：TurboMind 后端在某些中文场景和特定硬件（含国产卡）上很有竞争力，量化支持完善。

适用：国内部署、需考虑国产硬件，或重度依赖 InternLM 等模型的场景。

vLLM

优势：社区最活跃、文档最全、模型覆盖最广；支持 NVIDIA、AMD、Intel Gaudi、Google TPU、AWS Neuron 等；扩展性好，接新模型、新量化、新 attention 后端都容易。

劣势：NVIDIA 上极限吞吐通常略逊 TensorRT-LLM，前缀共享略逊 SGLang。

适用：默认首选。除非有压倒一切的特定需求，否则从 vLLM 起步几乎不会出错。

一句话：TRT-LLM 拼极限、SGLang 拼前缀、vLLM 拼通用。

09

真实世界中的应用

vLLM 是当下最流行的开源 LLM 服务引擎之一，对任何想在自家 GPU 上跑开源模型的团队几乎都是默认选项。两类场景尤其受益：

高流量聊天应用

连续批处理把每个 GPU slot 填满，PagedAttention 把所有用户的 KV cache 紧凑塞进显存，两头都不浪费。

Agent 系统

agent 每步都带同一段长指令（system prompt + 工具说明 + 历史对话），前缀共享省显存效果突出；而 agent 往往步骤多、单步回答短，连续批处理能把 GPU 空闲时间压到极低。

写 在 最 后

1

PagedAttention：把操作系统的分页思想搬到 KV cache，用固定小块按需分配显存，既消灭碎片浪费，又让相同前缀近乎免费地共享。
2

连续批处理：每个 decode 步动态放入新请求、放走已完成请求，让 GPU 永不空转。
3

OpenAI 兼容 API：现有代码几乎零成本接入。一个省显存、一个省算力，再裹一层标准接口，三者合到一起，同一张 GPU 就能服务远多于朴素方案的用户。

vLLM 不是这条赛道的唯一选手——极限吞吐有 TensorRT-LLM，前缀复用有 SGLang。

但凭着最活跃的社区、最广的模型与硬件支持，它依然是绝大多数团队最稳妥的默认起点。
