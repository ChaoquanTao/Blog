最近把一个 Kubernetes 集群从旧版本升级到了新版本，集群里跑着一个自研的 device plugin（以 DaemonSet 形式部署）。升级前一切正常，升级后这个 DaemonSet 一启动就疯狂报错：

// 报错日志

E  Device plugin register failed, err:
   rpc error: code = Unknown
   desc = failed to dial device plugin:
   context deadline exceeded

诡异的地方在于：代码一行没改，镜像还是同一个，只是集群版本变了。报错说的是"注册（register）失败"，错误内容却是"拨号（dial）超时"——到底是谁在向谁拨号？这篇文章就从 Device Plugin 的原理出发，把这个问题彻底讲清楚。

01 什么是 Device Plugin

Kubernetes 调度器天生只认识两种资源：CPU 和内存。但现实里节点上还有大量异构设备：GPU、RDMA 网卡、FPGA、TPU、加密卡……这些 Kubernetes 默认一无所知。

想让 Pod 能声明 nvidia.com/gpu: 2，就得有人告诉 kubelet 两件事：

① 节点上有哪些设备

这种设备有几个、健不健康。

② 怎么把设备塞进容器

某 Pod 申请设备后，如何挂设备文件、设环境变量等。

把硬件细节硬编码进 kubelet 不现实，于是 Kubernetes 提供了一个扩展点：Device Plugin 框架。这是典型的"机制与策略分离"。

基本架构

Device Plugin 本质上是一个 gRPC server，通常以 DaemonSet 部署到每个节点，与该节点的 kubelet 通过节点本地的 Unix domain socket 通信。

// 一个 Node 内部

kubelet

监听 kubelet.sock（注册服务）

▲ gRPC over unix socket ▼

device plugin（DaemonSet）

监听 cool.sock（自己的 gRPC server）

/var/lib/kubelet/device-plugins/

├── kubelet.sock  (kubelet 监听)
└── cool.sock     (插件监听)

关键约定

插件 Pod 要访问 kubelet 的 socket 目录，通常需要 hostPath 挂载该目录，并以 privileged 运行。

四个接口 + 一次握手

GetDevicePluginOptions

声明插件能力，一般返回空。

ListAndWatch

流式接口，持续向 kubelet 上报设备列表及健康状态。

Allocate

Pod 真正使用设备时被调用，返回要注入容器的环境变量、设备文件、挂载等。

GetPreferredAllocation

可选，给 kubelet 提供设备分配偏好。

注意一个容易忽略的细节：注册请求是插件主动发起的，但建立设备上报通道却是 kubelet 反过来拨号连插件。

// 注册握手时序（plugin ⇄ kubelet）

① plugin: Start()

listen cool.sock，让自己的 socket 可被连接。

↓

② plugin → kubelet: Register()

上报 endpoint、resourceName。

↓

③ kubelet → plugin: dial back

kubelet 反向连接 cool.sock 建立上报通道。本文 bug 就出在这一步。

↓

④ plugin → kubelet: ListAndWatch

上报设备列表，保持流。

↓

⑤ Register 返回成功

至此插件接入完成。

↓

⑥ Pod 调度后 → Allocate() → 注入容器

核心判断

插件必须先把自己的 gRPC server 起起来（socket 可被连接），kubelet 才能在注册时成功反向拨号。记住 ① 与 ③ 的关系。

最小骨架：注意 main 里的顺序

下面这段 main 的顺序，就是本文 bug 的核心：

func main() {
  p := NewPlugin()
  // 看似无害的顺序：先注册，再启动
  if err := p.Register(); err != nil { // ← 新集群会失败
    log.Fatalf("register failed: %v", err)
  }
  if err := p.Start(); err != nil {
    log.Fatalf("start failed: %v", err)
  }
  select{} // block forever
}

02 升级后遇到的奇怪问题

这份代码（先 Register() 后 Start()）在旧集群跑了很久都没问题。升级后，Pod 一启动就 CrashLoopBackOff。三个反直觉的点：

1

代码和镜像完全没变，唯一变量是集群版本——第一反应根本不会怀疑自己的代码。
2

错误显示在 Register()，内容却是 failed to dial device plugin——注册明明是插件主动发的，谁拨谁？
3

context deadline exceeded 是个超时——kubelet 在等一个永远等不到的连接。
03 基于原理的问题分析

定位：是谁在拨号

回到握手第 ③ 步：kubelet 处理注册时，会反过来连接插件声明的 socket。而我们的顺序是「先 Register，后 Start」，于是：

Register() → kubelet 收到，立刻去 dial cool.sock

↓

但 Start() 还没执行，cool.sock 根本不存在

↓

kubelet 拨号阻塞，超时

↓

Register() 收到 error：
failed to dial device plugin: context deadline exceeded

根因

错误里的 "device plugin" 指的就是我们自己的插件，是 kubelet 在抱怨"我连不上你"。socket 还没就绪，插件就抢先注册了。

那为什么旧集群一直没事

区别在于 kubelet 在不同版本里处理注册请求的方式。

旧版本 ≤ 1.24 · 异步回拨

注册 handler 收到 Register 后立即返回成功，把"反向拨号"丢进 goroutine 异步做。

go m.addEndpoint(r)    // 异步 fire-and-forget
return &Empty{}, nil // 立刻返回成功

Register 瞬间成功 → 插件继续 Start()、socket 建好 → 稍后 goroutine 才拨号，此时 socket 已就绪。错误顺序被"异步 + 时间差"悄悄兜住。

⇣ 重构后 ⇣

新版本 ≥ 1.25 · 同步回拨

goroutine 没了，改成在 handler 里同步连接插件，连不上就直接把 error 返回给注册请求。

if err := s.connectClient(...); err != nil {
  return &Empty{}, err // 同步拨号，失败即返回
}

connectClient 是带 grpc.WithBlock()、10 秒超时的阻塞拨号。插件没 listen，必然超时。

// 两个版本逐项对比

维度 ≤ 1.24 ≥ 1.25
回拨方式 异步 goroutine 同步 connectClient
何时返回 立即成功 拨号成功后
容错 时间差兜底 零容忍
现象 一切正常 注册即失败

版本、PR 与官方说法

来龙去脉

引入它的是 PR #109016——把 device-plugin 逻辑重构进独立 plugin 包，随 v1.25.0 发布。有意思的是，PR 本意只是重构，async → sync 的语义变化是个非预期副作用。

副作用后来才被发现（KubeVirt 升级后挂了），于是有了 issue #112395。维护者讨论后决定保留新语义：因为"先起 server 再注册"本就更正确、对新老集群都兼容。

官方文档与 1.26《Device Manager graduates to GA》博客把它确认为既定行为（代码实际落在 1.25，1.26 博客只是追认）。

修复：一行顺序调整

先 Start() 把 server 起好，再 Register()：

  p := NewPlugin()
- if err := p.Register(); err != nil { ... }   if err := p.Start(); err != nil { ... } + if err := p.Register(); err != nil { ... }   select{}

✓ 新集群（≥1.25）

kubelet 同步回拨时 socket 已就绪，握手成功。

✓ 老集群（≤1.24）

先起 server 从来没有副作用——它本就该这样。

04 写在最后

这个 bug 本身很小，调换两行代码就好，但它串起来的几件事值得记一下：device plugin 的握手是双向的，插件主动 Register、kubelet 却反向连接插件，所以正确顺序永远是先 Start 再 Register；旧版本 kubelet 的"异步注册"从来不是契约，只是实现细节，我们的代码顺序本就是错的，只是被它意外兜住——依赖外部系统时，要依赖它承诺的契约，而不是它当前的实现时序；而这次的导火索，恰恰是一次 async → sync 的改动，异步用时间差掩盖了顺序错误，同步则把它无情地暴露出来。

排查"代码没动、升级就挂"的问题，最高效的路径就是这篇文章的结构本身：

先理解原理（握手时序），再回到现象（谁在拨谁），最后到对方源码里找差异（async vs sync）。
