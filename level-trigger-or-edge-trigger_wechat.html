KUBERNETES · CONTROLLER 范式

level-trigger OR edge-trigger

从一个古老的触发哲学
读懂 Kubernetes controller 的开发范式

写给谁看

很多人从 kubebuilder 脚手架开始接触 controller，照着填 Reconcile 就能跑，但很快冒出一堆问号：为什么它不告诉我"发生了什么事件"？为什么要写得幂等？Finalizer 到底在防什么？这些问号都指向同一个源头。本文面向"会 Go、但没写过 controller"的工程师。

01 两种触发哲学：edge 与 level

"触发"这个词，最早的精确定义来自数字电路。一个寄存器要在什么时候采样输入信号？这里有两种截然不同的设计：

边沿触发 edge-triggered

在信号发生跳变的那一瞬间动作。关心的是"变化"这个事件本身——比如时钟从低到高的上升沿。

电平触发 level-triggered

只要信号当前处于某个电平就持续动作。关心的是"现在是什么状态"，而不关心它是怎么变成这样的。

这组概念后来被借用到各种软件系统里。最为 Linux 工程师熟悉的，就是 epoll 的两种模式：EPOLLET（edge）只在"有新数据到达"那一刻通知你一次，没读干净剩下的不会再提醒；默认的 level 模式只要缓冲区里"还有数据没读完"就会一直通知。

用一个更生活化的比喻把差别讲透。假设你要看护一个水箱，不让它溢出：

⚡ edge 的做法

在警戒线上装传感器，水面越过警戒线那一瞬间发信号。你收到信号就跑过去放水。

🔁 level 的做法

不装传感器，每隔一段时间走过去看一眼——"现在水位是不是高于警戒线了？高了就放水，没高就走。"

平时两种做法效果完全一样，差别要等到出意外时才显现。

02 为什么 edge-triggered 这么脆弱

edge-triggered 看起来更"高级"也更省事——有事它叫你，没事你睡觉。问题在于，它的正确性建立在一个极其脆弱的前提上：

"

每一个事件都必须被可靠地捕捉到，一个都不能漏。

而在真实世界里，这个前提永远不成立。我们看两个一定会发生的现实。

现实一 · 信号会丢

传感器可能接触不良抖一下、你正忙没收到、网络丢了包。无论哪种，那个"越线事件"就永久地消失了。水位一直高着，但"越线"这个瞬间已经过去，再也不会有第二个事件提醒你。水箱静悄悄地溢了。

现实二 · 进程会重启

放到软件里，进程会崩溃、会被滚动升级、会被驱逐重调度。在你离线这段时间里发生的所有事件，你全部错过了。等你回来，世界已经变样，而你手里没有任何"这期间发生过什么"的记录。

对 edge-triggered 来说这两个现实是致命的。它的整个工作模型就是"捕捉事件 → 响应事件"，一旦漏掉一环，系统就会永久地停在错误状态。翻译到业务代码里，就是这样一类 bug：

监听 "Pod 删除" 事件 → 收到后重建一个 Pod

demo 里跑得很好。但只要接收服务重启一次、恰好错过某个 Pod 的删除事件，这个 Pod 就永远不会被重建了。没有报错，系统只是悄无声息地少了一个副本——而且难以复现，因为本地不会恰好在那个时刻重启。

// 信号丢失时，两者命运对比

❌ edge-triggered

传感器发信号 → ✗ 信号丢了 → 之后再无新事件 → 水箱溢出

✅ level-triggered

巡检正常 → 巡检正常 → 巡检高了→放水 → 巡检正常 → 下一轮自动纠正

level-triggered 根本不在乎"刚才发生了什么事件"。它每一轮都重新观察当前的全量状态、重新计算现在该做什么。漏掉一万个事件都没关系——只要再巡检一轮，看到"水位还是高的"，就会再放一次水。

代价是你得不停巡检，这是"轮询的代价"。但换来一个无比珍贵的性质——系统能从任意错误状态中自愈。漏了事件下一轮纠正，进程重启起来重看全量，状态读错下一轮再读就对了。

任何一个需要在不可靠环境里保证最终正确的控制系统，最终都会收敛到 level-triggered 上。这不是品味问题，而是在一个会丢事件、会重启的世界里，唯一站得住脚的选择。

03 把 level-triggered 刻进基因的系统

Kubernetes 是一个声明式（declarative）系统：用户不写"怎么做"，只写"我要什么"。你提交一段 YAML 说"我要 3 个 nginx 副本"，这叫期望状态（desired state）；现在有几个、怎么变过去、节点挂了怎么补，都不归你管，系统自己想办法。

那"系统自己想办法"是谁在做？答案就是 controller（控制器）。

Controller 就是那个不断观察当前状态、和期望状态比较、采取行动消除差异的循环。它有一个更标准的名字：控制循环（control loop）。

// 所有控制循环的形状都一样

观察 → 比较 → 纠偏

这些控制循环跑在一个会丢事件、会重启的分布式环境里：API Server 会重启，controller 会被升级重调度，watch 长连接会断开重连。要保证"最终正确"，答案我们已经推过了——它必须是 level-triggered 的。

"

不要响应事件，而要调谐状态（reconcile）。永远拿当前的全量状态去和期望比较，把差异消除掉；不要去关心"刚才发生了哪个事件"。

先澄清一个误解

controller 明明靠 watch 监听对象变化工作，那它不就是 edge-triggered 吗？

watch 事件只是性能优化，不是决策依据。它只是敲一下你的肩膀："default/my-app 这个对象可能变了，你该去看一眼了。"收到提醒后，controller 并不处理这个事件，而是拿着名字重新读取它当前的完整状态、重新调谐一遍。事件丢了也没关系，还有定期全量重新同步（resync）兜底。

事件用来触发，状态用来决策。

04 范式落地：七个要点

挂在主线上的七个要点，每一个要么是范式逼出来的硬约束，要么是 Kubernetes 为了让你更容易遵守它提供的便利。

1

Reconcile：为什么它只接收一个 key

用 controller-runtime 框架，你要写的核心就是一个 Reconcile 函数。最值得玩味的是入参 ctrl.Request，它里面只有一样东西：对象的 key，也就是 namespace/name。

func (r *FooReconciler) Reconcile( ctx context.Context, req ctrl.Request, ) (ctrl.Result, error) // req 里只有 {Namespace, Name}，没有事件类型

它不接收事件类型。没有"这是 Create 还是 Update 还是 Delete"，框架只告诉你一句话——"default/my-app 这个对象，你该看一眼了。"这正是 level-triggered 在 API 层面的直接体现：框架不告诉你"发生了什么"，只告诉你"该看谁"。

// reconcile 的标准四步骨架

① 读期望对象

重新观察当前全量状态（level-triggered）

↓

② 处理删除 / finalizer

对象正在被删？先做临终清理

↓

③ 确保子资源存在

ensure 而非 create（幂等）

↓

④ 回填 status，向用户汇报实际状态

2

幂等 idempotency：被 level-triggered 逼出的第二条铁律

同一段逻辑，对同一个状态，跑 1 次和跑 100 次，结果必须完全一致。

既然 controller 每一轮都重新观察、重新执行，那么同一对象的 reconcile 逻辑会被反复地、不确定次数地执行：风平浪静时每 10 分钟 resync 一次，状态频繁变化时 1 秒触发好几次，崩溃重启后把所有对象从头跑一遍。如果你写成"每跑一次就创建一个 Pod"，100 次 reconcile 就会创建 100 个 Pod——灾难。

// 幂等写法：ensure 一个符合期望的 Pod

不存在 → 创建 已存在但不对 → 更新 已存在且正确 → 什么都不做

一枚硬币的两面

level-triggered 决定了"逻辑会被重复执行"，幂等保证了"重复执行不会出乱子"。

3

Informer / Cache：每轮都读全量，API Server 受得了吗

先交代 API Server 是什么：它是整个集群唯一的入口。所有对象都存在一个叫 etcd 的数据库里，谁都不能直接碰 etcd，所有读写都必须经过 API Server 这道门。kubectl、kubelet、controller 全都是它的客户端。

kubectl ──┐ controller ┼──► API Server ──► etcd kubelet ──┘ （唯一入口/门卫） ◄── （真正存数据） 所有读写都得走这道门，etcd 不对外直接开放

最朴素的实现是每次 reconcile 都向 API Server 发 GET/LIST。但API Server 扛不住：几十个 controller、成千上万个对象，每次都去打它，会被读请求压垮。解决方案是 Informer（背后是本地 Cache），机制是经典的 List + Watch：

① List

启动时拉一次某类资源的全量列表，灌进本地内存缓存。

② Watch

建立长连接，API Server 把后续增量变化持续推过来，实时更新本地缓存。

③ 读缓存

reconcile 里读对象时读的是本地缓存，根本不碰 API Server。

这就把"成千上万次读 API Server"压缩成了"一次 List + 一条 Watch 长连接"。

代价 · 短暂陈旧 stale

本地缓存可能短暂陈旧——API Server 上对象已变，但 Watch 事件还在路上。但读到旧数据无所谓：缓存马上会被更新，更新又触发一次 reconcile，下一轮就读到新值。陈旧状态会被下一次 reconcile 自愈——这正是 level-triggered + 幂等大显身手的地方。

4

WorkQueue：把"惊群"压平

惊群（thundering herd）：一件事发生时一大群对象同时被惊动、一拥而上——就像往鸽群里扔面包，所有鸽子哗地全飞过来。典型场景有两类：某个对象极短时间内被改几十次；或 controller 刚启动/刚重连时，缓存里成千上万个对象"看起来全都变了"。Informer 收到变化后并不直接调用 Reconcile，而是先把 key 塞进 WorkQueue，中间隔这一层是为了三件事——而这三件事只有 level-triggered 才敢这么干：

A

去重 dedup。同一对象短时间变 50 次，不会触发 50 次。队列维护一个 dirty 集合，正在处理时新事件只把它标为 dirty，处理完再重跑一轮。敢这么合并，正因为 level-triggered 根本不关心那 50 个事件分别是什么。

B

限速 rate limiting。控制 reconcile 速率，防止某个频繁抖动的对象把 worker 和 API Server 压垮。

C

失败重试 + 指数退避。reconcile 返回 error 时框架把 key 重新入队，退避时间逐次翻倍（示意：1s→2s→4s…）。既保证"出错的对象最终会被重试到成功"，又不会持续失败时把 API Server 拖垮。

你的 Reconcile 只要"返回 error"就行——重试、退避、去重，框架全替你兜住了。

5

OwnerReference：级联删除与反向触发

controller 通常会为它管理的对象创建子资源（比如 Foo controller 为每个 Foo 创建一个 Deployment）。两个需求都靠 OwnerReference（属主引用） 解决：

级联垃圾回收 cascading GC

GC 器看到"属主 Foo 没了"，会自动回收所有以它为 owner 的子资源。你不用写一行删除代码。

反向触发（controller-runtime 里叫 Owns）

声明 Owns(&Deployment{}) 后，某个带 Foo 属主引用的 Deployment 变了（比如被人删了），框架就把它属主 Foo 的 key 塞进 WorkQueue，触发 Foo 的 reconcile。

子资源被人动了 → 触发属主重新 reconcile → 属主重新观察"子资源还在不在、对不对" → 不对就修。你不需要写"监听删除事件然后重建"这种脆弱逻辑，只需在 reconcile 里幂等地 ensure 子资源存在。

6

Finalizer：删除前的临终清理

级联 GC 能回收集群内的子资源。但如果你的 controller 在集群外也分配了东西呢——云厂商的负载均衡器、外部数据库的一条记录、对象存储的一个 bucket？用户删对象时 GC 管不到这些，对象一从 etcd 消失，你就再也没机会释放它们了。Finalizer（终结器） 就是为此而生的"临终清理"钩子：

① 用户 delete

deletionTimestamp 被设置，对象进入 Terminating 但不消失

↓

② reconcile 检测到 deletionTimestamp != nil

执行外部资源清理（释放 LB / 删记录 / …）

↓

③ 移除自己的 finalizer

清理成功后把标记摘掉

↓

finalizers 空了 → Kubernetes 真正删除对象

这是 controller 里最容易出 bug 的地方，原因还是那两条铁律：

✗ 清理必须幂等

reconcile 可能在"清理"和"移除 finalizer"之间崩溃重启，重启后又重进清理逻辑。"释放外部资源"必须能安全跑第二次。

✗ 对 NotFound 宽容

要删的外部资源"本来就不存在了"不能当成错误，否则 finalizer 永远移不掉，对象永远卡在 Terminating。正确写法：删，如果返回 NotFound，当作成功。

7

Status 与 Conditions：对外的单一事实源

spec 和 status 的分工正好对应控制循环的两端：

spec · 期望状态

由用户写，是控制循环的输入。用户只动 spec。

status · 实际状态

由 controller 写，是控制循环的输出。用户只读不写。

// kubectl get foo my-app -o yaml（节选）

metadata: generation: 2 # 改一次 spec +1 spec: # 用户写：我想要什么 replicas: 3 image: nginx:1.27 status: # controller 写：实际怎么样 observedGeneration: 2 # 已处理到 gen=2 conditions: - type: Ready status: "True" # True/False/Unknown reason: AllReplicasReady message: "3/3 replicas available"

三个要点

① status 是对外的单一事实源。别人想知道"对象现在怎么样"只看 status，常用结构化的 Conditions（type / status / reason / message）表达健康与进度。

② observedGeneration 判断 status 是否跟上最新 spec。observedGeneration < generation 说明 controller 还没处理你最新的改动。

③ spec 与 status 走不同子资源更新。更新 spec 用 Update，更新 status 用 Status().Update()，两者互不覆盖——保证"用户意图"与"controller 汇报"井水不犯河水。

05 三条纪律：先在脑子里装一句话

"

不要问"发生了什么，我该做什么反应"；要问"现在是什么样，期望是什么样，我怎么把前者变成后者"。

每当你想写下 if 收到删除事件 { ... } 时，停下来——这是 edge-triggered 思维在作祟。正确姿势永远是：读全量、算差异、消差异。展开就是三条纪律：

1

决策只依赖"当前读到的状态"，绝不依赖"我是被什么触发的"。Reconcile 从头到尾都不该关心自己为什么被调用。

2

每个写操作都是"ensure"，不是"do once"。创建、更新、删除，都要能安全地重复执行——因为它一定会被重复执行。

3

把每一种中途崩溃都想一遍。"如果我在这一行之后、下一行之前挂掉，重启后重进 reconcile，会出乱子吗？"如果会，说明这段还不够幂等。

06 骨架代码：把七个要点串起来

ctrl.NewControllerManagedBy(mgr). For(&v1.Foo{}). # 主资源 Foo 变了→入队 Owns(&appsv1.Deployment{}). # 子资源变了→反向触发属主 Complete(&FooReconciler{...}) # 声明"看谁"，而不是"处理什么事件" # 生产必开 LeaderElection：只有 leader 跑 reconcile

// Reconcile 主干：四步骨架

// ❶ 读期望对象（level-triggered） var foo v1.Foo if err := r.Get(ctx, req.NamespacedName, &foo); err != nil { return ctrl.Result{}, client.IgnoreNotFound(err) // 对 NotFound 宽容 } // ❷ 处理删除 / finalizer if !foo.DeletionTimestamp.IsZero() { if ContainsFinalizer(&foo, finalizer) { if err := r.cleanupExternal(ctx, &foo); err != nil { return ctrl.Result{}, err // 失败→指数退避重试 } RemoveFinalizer(&foo, finalizer) r.Update(ctx, &foo) } return ctrl.Result{}, nil } // ❸ 确保子资源存在（ensure 而非 create，整段幂等） if err := r.ensureDeployment(ctx, &foo); err != nil { return ctrl.Result{}, err } // ❹ 回填 status，走 /status 子资源 foo.Status.ObservedGeneration = foo.Generation r.Status().Update(ctx, &foo) return ctrl.Result{}, nil

整份代码里找不到一处"监听某事件 → 做某动作"的影子。所有逻辑都是"读当前状态 → ensure 成期望的样子"。这就是范式本身。

写在最后

回到开篇的二选一：level-trigger OR edge-trigger? 对 Kubernetes controller 来说，这根本不是一道选择题。在一个会丢事件、会重启的世界里，edge-triggered 那种"捕捉事件 → 响应事件"的模型天生脆弱——漏一个事件就可能永久停在错误状态；而 level-triggered "每轮重读全量、重算差异"能从任意错误中自愈。

后半篇讲的所有东西——Reconcile 只收 key、幂等、Informer/Cache、WorkQueue、OwnerReference、Finalizer、Status/Conditions——没有一个是孤立的"知识点"。它们要么是 level-triggered 逼出的硬约束，要么是 Kubernetes 为了让你更容易遵守它铺的便利设施。

真正要记住的只有一句话：
不要响应事件，要调谐状态。

把它装进脑子，你再看任何 controller 源码，
都能立刻问出那个对的问题——
它在把什么 actual 调谐到什么 desired？
